# 要件仕様書: アプリ起動時のトークン有効性検証

## 1. 目的

アプリ起動時に SecureStore に保存されたトークンの有効性をバックエンドに問い合わせ、無効なトークンではログイン状態に遷移しないようにする。

現在、`initialize()` は SecureStore にトークンが「存在する」だけで `isAuthenticated: true` に設定している。そのため、トークンが期限切れやサーバー側で無効化されていても、アプリがログイン済み画面（Settings画面の Sign Out ボタン、プロフィール情報など）を表示してしまう。

既存の TanStack Query の 401 ハンドラー（`apps/mobile/lib/providers.tsx`）は、ログイン後の通常利用中に 401 を検知して `clearAuth()` を呼ぶ安全網として機能している。しかし、`initialize()` から最初のクエリが発火するまでの間にギャップがあり、その間ユーザーは無効な認証状態でアプリを操作できてしまう。本修正はこのギャップを閉じるものであり、既存の実行時保護を置き換えるものではない。

## 2. ユーザーストーリー

アプリ利用者として、アプリを起動したときに無効なトークンでログイン済み画面が表示されないようにしたい。なぜなら、無効な認証状態でアプリを操作すると、データが取得できずに壊れた画面が表示され、混乱するから。

## 3. 機能要件

### 3.1 initialize() のトークン検証フロー

`initialize()` は SecureStore からトークンを取得した後、`authenticatedRequest` 経由で `me` クエリをバックエンドに送信し、トークンの有効性を検証する。

検証結果に応じて3つの分岐がある:

- **成功**: `me` クエリが正常に応答を返した場合、`isAuthenticated: true` に設定し、アプリのメイン画面へ遷移する
- **認証失敗（401）**: `authenticatedRequest` 内でアクセストークンの 401 検知 → `refreshAuth()` によるリフレッシュトークンでの再取得が自動的に試みられる。リフレッシュも失敗した場合、`clearAuth()` を呼んでトークンを削除し、ログイン画面へリダイレクトする
- **ネットワークエラー**: 接続できない場合、リトライ画面を表示する

### 3.2 リトライ画面

- ネットワークエラー時に表示する
- 既存の `ErrorScreen` コンポーネントを再利用する
- メッセージは「ネットワーク接続を確認してください」（i18n 対応）とする
- リトライボタンを押すと `initialize()` のフロー全体を再実行する
- リトライの結果もまた3分岐で処理される（成功→メイン画面、401→ログイン画面、ネットワークエラー→リトライ画面のまま）

### 3.3 ローディング表示

- 検証中は既存の `LoadingScreen`（スピナーのみ）をそのまま表示する
- 追加のメッセージ表示は行わない

### 3.4 状態表現

アプリの `_layout.tsx` レベルで初期化の3つの結果を区別できる必要がある:

- **ローディング中**: `LoadingScreen` を表示（現状と同じ）
- **成功 / 認証失敗**: `NavigationGuard` が `isAuthenticated` に基づいてメイン画面またはログイン画面にルーティング（現状と同じ）
- **ネットワーク到達不可**: `ErrorScreen` によるリトライ画面を表示（新規）

### 3.5 SecureStore にトークンが存在しない場合

現状と同じ動作を維持する。トークンがなければネットワーク通信は行わず、即座に `isAuthenticated: false` となり、ログイン画面へ遷移する。

### 3.6 既存の refreshAuth() リトライの尊重

`refreshAuth()` は既にネットワークエラーに対して指数バックオフ付きの最大4回リトライを内蔵している。リトライ画面はこの組み込みリトライがすべて失敗した後にのみ表示される。短時間の接続断ではリトライ画面は表示されない。

## 4. 非機能要件

- **起動時間**: ネットワーク通信が追加されるため、initialize() の実行時間が増加する。ただし、良好なネットワーク環境では `me` クエリは軽量であり、体感的な遅延は最小限に抑える
- **セキュリティ**: 無効なトークンでログイン済み画面に到達することを防止する。ネットワーク到達不可の場合もログイン状態にはしない（厳格方針）
- **冪等性**: リトライボタンの連打に対して安全であること。複数の `initialize()` が同時に実行されないようにする

## 5. 制約事項

- **モバイルアプリのみの修正**: API 側のコード変更は行わない。既存の `me` クエリ、JWT 検証ミドルウェア、`authenticatedRequest` の 401 ハンドリングをそのまま利用する
- **既存の 401 ハンドリングとの整合**: `providers.tsx` の TanStack Query 401 ハンドラーは変更しない。通常利用中の安全網として引き続き機能する
- **既存テストの更新**: `apps/mobile/stores/auth-store.test.ts` の既存テスト（特に `initialize` のテスト2件）は新しい動作に合わせて更新する必要がある

## 6. スコープ外

- **API 側の変更**: バックエンド（Rust API）のコード変更は含まない
- **display_name 修正**: 前セッションで作業中の `apps/api/src/auth/service.rs`、`custom_mutations.rs`、`user_service.rs` の変更（サインアップ時の display_name 付きユーザーレコード即時作成）は別件として扱う
- **トークンのローカル有効期限チェック**: JWT のペイロードをデコードして `exp` をローカルで検証する方式は採用しない（サーバー側で無効化されたトークンを検知できないため）
- **バックグラウンドでの定期的なトークン検証**: アプリがフォアグラウンドに復帰したときの再検証は含まない
- **オフラインモード**: ネットワーク接続がない状態でアプリを使用可能にする機能は含まない

## 7. 受け入れ基準

1. SecureStore に有効なトークンがある状態でアプリを起動すると、`me` クエリでバックエンドにトークン有効性を確認した後、メイン画面（tabs）に遷移する
2. SecureStore に期限切れのアクセストークンがあるが、リフレッシュトークンが有効な場合、自動的にトークンが更新され、メイン画面に遷移する
3. SecureStore にアクセストークンとリフレッシュトークンの両方が無効な状態でアプリを起動すると、SecureStore のトークンが削除され、ログイン画面にリダイレクトされる
4. ネットワーク接続がない状態でアプリを起動すると、リトライ画面（「ネットワーク接続を確認してください」メッセージとリトライボタン）が表示される
5. リトライ画面でリトライボタンを押すと、検証フロー全体が再実行される
6. リトライ後にネットワークが復旧していれば、検証結果に応じてメイン画面またはログイン画面に遷移する
7. SecureStore にトークンがない場合は、ネットワーク通信を行わずにログイン画面に遷移する（既存動作を維持）
8. 検証中は LoadingScreen（スピナー）が表示される
9. 既存の TanStack Query 401 ハンドラーの動作が変わらない
10. 既存のテストが新しい動作に合わせて更新されている
